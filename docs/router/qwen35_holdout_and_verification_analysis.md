# Qwen3.5 Holdout and Verification Analysis

M11b evaluates the unadapted `Qwen/Qwen3.5-4B` on the same 30-row holdout used
for LoRA v001-small. Both runs use prompt-router v2, the same five few-shot
examples, schema repair enabled, 256 maximum new tokens, temperature zero, and
local model files only. The only model difference is LoRA adapter loading.

LoRA changed one measured routing decision beneficially. On holdout case 010,
base selected `request_more_info` while the expected mode was
`external_expert`; LoRA selected `external_expert`. The other base mode misses,
cases 020 and 022, remained. Tools, risk, fusion, and strict verification
metrics did not improve or regress.

The evidence supports a narrow conclusion: v001-small improved one mode
boundary, but it did not demonstrate broader holdout improvement.

## Verification Vocabulary

The holdout defines 25 expected verification labels. Base emitted 49 unique
labels and LoRA emitted 48; every emitted label was outside the exact expected
vocabulary. Consequently, both runs scored 0/30 for strict containment.

The mismatch has three causes:

1. **Representation mismatch.** Predictions use natural-language labels such
   as `citation check`, `dimension consistency`, and
   `tangent consistency check`. The eval requires `citation_check`,
   `dimension_check`, and `tangent_consistency_check`.
2. **Prompt conflict.** `prompts/router_fewshot_v2.jsonl` explicitly teaches
   natural-language checks such as `symmetry check`,
   `collect independent judgments`, and `request missing file`. The system
   prompt requires concrete checks but does not require canonical snake_case
   identifiers.
3. **Semantic granularity mismatch.** Predictions often choose task-specific
   observations such as `Jacobian determinant check`, `log-log slope fit`, or
   `missing mesh study`, while the eval expects broader policy checks such as
   `dimension_check`, `convergence_check`, or `safety_approval_check`.

Frequent examples:

| Predicted | Count per run | Canonical target |
| --- | ---: | --- |
| `citation check` | 4 | `citation_check` |
| `dimension consistency` | 3 | `dimension_check` |
| `tangent consistency check` | 1 | `tangent_consistency_check` |
| `sign convention check` | 1 | `sign_convention_check` |
| `unit test run` | 1 | `run_tests` |

The complete per-label and per-case audit is generated by:

```bash
python scripts/analyze_verification_labels.py \
  evals/router_eval_holdout_001.jsonl \
  eval_results/qwen35_4b_holdout_eval_001.json \
  eval_results/qwen35_lora_v001_small_holdout_eval_001.json
```

Alias suggestions from this script are lexical heuristics for review, not an
automatic scoring equivalence.

## v002 Priorities

Before approving v002 training:

1. Replace natural-language verification labels in prompt-router few-shot
   examples with the canonical identifiers in
   `docs/router/router_canonical_vocabulary.md`.
2. State in the system prompt that `verification.checks` accepts canonical
   lowercase snake_case identifiers only.
3. Normalize all SFT assistant outputs and eval expectations to the same
   vocabulary; preserve explanatory detail in `verification.reason`.
4. Add diverse examples for contact/nonlinear/code risk calibration because
   both base and LoRA underestimated 8/30 holdout cases.
5. Run prompt-only base and v001-small holdout evals after vocabulary repair
   before deciding whether another weight update is justified.

No additional training, LoRA dry-run, adapter update, or API access occurred
in M11b.
